Vivado Build Scripts

Avioding the Xilinix Vivado User Interface by building projects using scripts and makefiles

Mar 28, 2025

When working with the ice40 and ECP5 devices, where the FPGA images i’m creating are built using command line tools, they lend themselves to building scripts and makefiles to generate them. While I use Yosys and Next-PNR with the lattice devices for my personal projects over the last few years, in my day job i’m much more exposed to the Xilinix Devices. As a result I have found myself wanting to experiment with the low end versions of these devices but the tutorials found online will always push towards using the GUI to create and build projects, which is a big change to my personal workflow.

To experiment with a purchased two Micrphase Devices, one based on the arty and and other on a small Zynq, these are both compatible with the current open source tools so I can use them this way, but I also wanted to look at using the Xilinix Tools, for this it was time to venture into Tcl scripts to see if I could create a workflow similar to my other personal projects.

An example build tcl file, which allows for the vivado batch script to be used to read the tcl file building the project and creating the bitfile, this is based on Projectf build script post which I found a very useful starting point.

I made a simple project for a recently purchased Artix based MicroPhase development Board, which I wanted to build a simple example for. This project is designed to flash the LED on the board.

# read design sources (add one line for each file)
read_verilog -sv "top.v"

# read constraints
read_xdc "board.xdc"

# synth
synth_design -top "top" -part "xc7a35tfgg484-1"

# place and route
opt_design
place_design
route_design

# write bitstream
write_bitstream -force "hello.bit"

With the tcl file above I can call vivado from the command line using vivado -mode batch -source build.tcl' which runs the script and generates the hello.bit` bitstream for my FPGA. By using this structure, and placing a pin out file at the top layer, i’m able to build a couple of scripts that reuse the same source code but are used by different tools. I good use of this could be building and testing with both vivado, and having a second set that use yosys/nextpnr with the same code.

To enable more advanced setups and automation the next set is running all this though a makefile, this needs to create the tcl file that will be used in vivado itself, so that the intermediate step is always upto date and the build process is all generated and called from within the project.

# FPGA settings
PROJECT_NAME = blink
FPGA_PART = xc7a35tfgg484-1
FPGA_TOP = top

#Files for Synthesis
SYN_FILES = top.v

# XDC Files
XDC_FILES = board.xdc

default: program
build: $(FPGA_TOP).bit

%.bit:
        echo "read_verilog -sv top.v" > program.tcl
        echo "read_xdc $(XDC_FILES)" >> program.tcl
        echo "synth_design -top $(FPGA_TOP) -part $(FPGA_PART)" >> program.tcl
        echo "opt_design" >> program.tcl
        echo "place_design" >> program.tcl
        echo "route_design" >> program.tcl
        echo "write_bitstream -force $(PROJECT_NAME).bit" >> program.tcl
        vivado -nojournal -nolog -mode batch -source program.tcl

.PHONY: check
check:
        verilator --lint-only -cc top.v

.PHONY: test
test: 
        verilator -Wall -GWIDTH=3 --trace --build --coverage -cc --exe top.v test.cpp
        ./obj_dir/Vtop
        verilator_coverage logs/coverage.dat --annotate logs/annotated --write-info logs/coverage.info --write logs/results.txt --rank
        genhtml logs/coverage.info --output-directory logs/html
        lcov --list logs/coverage.info > logs/coverage.txt

program: $(FPGA_TOP).bit
        openFPGALoader -c ft232 $(PROJECT_NAME).bit

.PHONY: clean
clean:
        rm program.tcl $(PROJECT_NAME).bit

The echo commands do the work of generating out tcl script, using the varriables defined at the top of the Makefile, this includes the FPGA details but also the pin definition file. The Makefile also inludes some of the testing and verilator checking that i use as part of the development process. Allowing for all the standard building commands to be captured here, even including using openFPGALoader to be able to deploy the generated bit file to the development board, in this case using the FT232 programmer that is built into the board.

With these simple integrations it is now possible to automate large parts of the build process and avoid having to open the Vivado, and hopefully with a bit more work be able to support some more advanced projects.
